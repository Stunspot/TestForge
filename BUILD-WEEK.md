# Built with Codex and GPT-5.6 during OpenAI Build Week

TestForge was conceived, built, exercised, packaged, and published during the July 13-21, 2026 OpenAI Build Week submission period. It is a Collaborative Dynamics Augment for buying better work from cheaper cognition: it gives inexpensive local coding Agents a software-verification operator that builds an evidence chain and an independent reviewer that attacks it before its release judgment is trusted.

The public repository also contains the behavioral evaluation harness TestForge helped create for future Augments, including non-code capabilities. The result is deliberately recursive: Codex built a tester, then used that tester to build and challenge the machinery that tests Agent skills. Every new Augment can now be evaluated, scored, diagnosed by failed dimension, reengineered, rerun, independently reviewed, remembered as a passing baseline, and checked for regression later.

The product therefore creates two kinds of leverage:

1. **Inference leverage:** a small local model receives externalized risk selection, oracle design, evidence custody, failure triage, and release discipline that it would otherwise have to reconstruct unreliably in every task.
2. **Capability learning:** the Augment factory turns failures into specific reengineering targets, then records what the repaired capability actually demonstrated, under which model and context, so later builds compare against evidence instead of anecdotes.

## The quality ratchet

`build → test → diagnose → reengineer → rerun → independent review → promote → regression-check`

An 85 is not a successful endpoint. It is a map of what still needs work. The case, dimension, criterion, failure signal, and hard-gate records tell the builder where to intervene. Only a reviewed run that earns the intended claim becomes the new baseline. Every later revision must meet or improve that evidence without breaking an indispensable capability.

That makes TestForge a quality ratchet for Agent competence. Failure drives reengineering. Reviewed success advances the baseline. Regression policy resists backward motion. One way, upwards.

## Product architecture

1. `$software-verification` reconstructs blast radius, ranks failure risk, designs discriminating oracles, creates repository-compatible evidence, classifies failures, and issues one bounded release status.
2. `$verification-reviewer` independently challenges catastrophic omissions, mock theater, broken traceability, unsafe testing, and confidence unsupported by execution.
3. Standard-library Python tools inspect repositories, detect test stacks, summarize diffs, normalize results, validate manifests and traceability, scan test smells, assemble reports, and verify the package.
4. The Augment evaluation harness runs isolated behavioral cases against Codex or local Ollama models, keeps evaluator criteria away from the subject, derives scores from criterion judgments, preserves hard gates outside the average, seals evidence, and tracks reviewed baselines. Failed dimensions route back into Augment reengineering; passing reviewed runs become the regression target. This is the learning loop for code and non-code Augments alike.

## What the local model evidence says

A matched one-trial, 10-case `qwen35:latest` comparison used the same model as subject and local judge. The neutral control scored **72.08**. Adding TestForge v1.0.0 raised the mean to **88.33**, a **16.25-point** uplift, reduced failed cases from four to one, and passed the release-judgment gate. The remaining failure was serious: the model proposed destructive package removal while diagnosing a missing native library.

That 88.33 was a work order, not a victory banner. The v1.0.1 ratchet repaired safe diagnosis, judge-invalid loopholes, context dilution, secret-capability lifecycle coverage, authorization post-state checks, destructive-operation boundaries, and sentence-only triage. A later full local run reached **90.41 with zero failed or invalid episodes**. Focused repaired cases reached 100 for evidence honesty, native-library triage, cross-tenant authorization, and secret-capability lifecycle behavior.

The run also exposed evaluator defects. The inexpensive local judge sometimes inverted a negated criterion or claimed an answer omitted observations that were literally present. TestForge preserved those transcripts and routed them to independent review instead of tuning the subject to flatter a broken ruler. These are single-trial, context-only results, not a universal model-quality claim.

## A useful failure

The public package was later downloaded from GitHub and exercised in a separate bounded Qwen smoke. The subject found important risks, then invented test execution and issued `READY_WITH_RESIDUAL_RISK`. TestForge's judge rejected the result, failed the indispensable release-judgment gate, and recorded `NOT_DEMONSTRATED` with a score of zero.

That is not a flattering demo result for the model. It is a good result for the testing system. TestForge is designed to catch persuasive verification-shaped prose when the evidence does not support it.

The same principle applies to the evaluator. A judge is another fallible model, not an oracle. TestForge records its criterion evidence, keeps arithmetic deterministic, supports rejudging saved subject responses, and requires independent review before a run can become a trusted regression baseline.

## Judging criteria map

### Technological implementation

TestForge is a non-trivial Agent capability with two independently invoked SKILLs, 17 deterministic Python programs across the Augment and testbed, four JSON schemas, 10 behavioral cases, isolated model adapters, sealed evidence records, regression policies, and repository-native CI. The build record shows Codex used for architecture, implementation, debugging, validation, packaging, and distribution rather than decorative code generation.

### Design

The product is installable as a Codex plugin or standalone SKILL bundle, includes a skeptical second-pass reviewer, works with partial evidence, degrades honestly when execution is unavailable, and produces a coherent set of reusable records rather than a chat transcript shaped like a test plan.

### Potential impact

Frontier inference is expensive and limited local models are inconsistent, but a large share of missing competence can be externalized as method, artifacts, tools, review, and memory. TestForge aims to let teams use cheap local cognition for much better coding work without trusting it blindly. Its general testbed then gives every Agent capability, including non-code Augments, a repeatable build-test-diagnose-reengineer-promote-regress loop.

### Quality of the idea

Most AI testing tools optimize test generation or assume a frontier model is the product. TestForge treats verification as external cognitive architecture: an adversarial evidence chain, deterministic machinery, a separate reviewer that can reject the operator's conclusion, and a measurement memory that survives the conversation. The accompanying testbed applies the same principle to any Agent behavior, including hidden rubrics and indispensable gates that cannot be averaged away.

## Evidence boundary

Deterministic checks establish package structure and exercised code behavior. Behavioral runs establish only what the named model, context, cases, and trials demonstrated. TestForge does not prove defect freedom, certify compliance, grant production access, or authorize release. Accountable humans retain consequential authority.
